## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline padding (`gap-1`, `p-1`) |
| sm | 8px | Compact element spacing (`gap-2`, `p-2`) |
| md | 16px | Default element spacing, form field gaps (`gap-4`, `p-4`) |
| lg | 24px | Section padding, card bodies (`gap-6`, `p-6`) |
| xl | 32px | Layout gaps between sections (`gap-8`) |
| 2xl | 48px | Major section breaks (`py-12`) |
| 3xl | 64px | Page-level spacing (`py-16`) |

Exceptions:
- Touch targets (buttons, icon-only controls): minimum 44px height on mobile. Use `py-3` (12px top + 12px bottom + font height) which meets this for md/lg size buttons.
- Modal footer button rows: `gap-2` (8px) — consistent with standard sm spacing token.
- Form input internal padding: `px-4 py-2` — 16px horizontal, 8px vertical.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#fafaf9` light / `#0c0a09` dark | Page background (`--color-bg-primary`) |
| Secondary (30%) | `#ffffff` light / `#1c1917` dark | Cards, modals, form surfaces (`--color-bg-surface`) |
| Accent (10%) | `#d97706` light / `#f59e0b` dark (amber) | Primary action buttons, focus rings, active borders, Retry button |
| Destructive | `#dc2626` light / `#ef4444` dark (red) | Delete buttons, error messages, danger badge variant |

Accent reserved for:
1. `Button variant="primary"` — only the affirmative CTA in each modal/form
2. Focus rings on Input, Textarea, Select fields (`focus:ring-amber-400`)
3. Border highlight on interactive Card hover (`hover:border-amber-400`)
4. Retry button on AI error state (inline, amber text)
5. Link-style empty state actions (amber text, no background)

Not for: secondary actions, cancel buttons, nav items, badges, general text.

Dark mode note: All token values have established dark mode counterparts in `globals.css`. Executor must use `dark:` variant classes throughout, consistent with existing components.

## Interaction Patterns

### AI Output Persistence (STAB-01, STAB-02)

Packing list and meal plan pages must implement this pattern:

1. **Initial state (no saved output):** Show EmptyState with emoji, title "No packing list yet", body "Generate one to get started." CTA button: "Generate Packing List" (amber primary).
2. **Saved state (output exists):** Show the result. Above the list, show a muted metadata line: "Generated [relative time]". Below the metadata, show a ghost Button: "Regenerate". No confirmation needed for regeneration (D-03 — one list per trip, no history).
3. **Loading state (generating):** Replace the CTA or Regenerate button with a loading spinner (Button `loading={true}`). Show inline copy: "Generating..." — no full-page skeleton.
4. **Error state:** Replace the result area with an inline error message. Copy: "Couldn't generate list — Claude returned an unexpected response. Tap Retry to try again." Show a "Retry" Button (`variant="ghost"`, amber text) inline below the error message. No modal, no navigation.

### Malformed Claude Response Error (STAB-01, D-12, D-13)

- Error appears inline where the AI result would render
- Never navigates away or shows a blocking modal
- Always offers a Retry action
- Status 422 from API → show the error + retry UI in the client

### Delete Confirmations (STAB-03, STAB-04, STAB-05)

All destructive actions use `ConfirmDialog`. Never use `window.confirm()`. Pattern:

```
title: "Delete [item name]?"
message: "[Consequence statement]"
confirmLabel: "Delete"
confirmVariant: "danger"
```

See copywriting section for specific copy per action.

### Loading States

- Form submits: Button `loading={true}` — spinner replaces icon, button disabled
- AI generation: Button `loading={true}` inline with copy "Generating..."
- Delete in progress: ConfirmDialog confirm button shows "Deleting..."
- Never use full-page spinners for operations that complete in < 5s
