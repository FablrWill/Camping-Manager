  <name>Task 2: Create POST /api/trips/[id]/review batch endpoint</name>
  <files>app/api/trips/[id]/review/route.ts, tests/review-batch.test.ts</files>
  <behavior>
    - REV-03: gear items in gearUsage array have their PackingItem.usageStatus updated
    - REV-03: gear items NOT in gearUsage array are untouched (no updateMany wipe)
    - REV-04: mealFeedbacks entries are upserted to MealFeedback (find existing → update, else create)
    - REV-05: spotRating writes to Location.rating when trip.locationId is not null
    - REV-05: spot update is skipped (no error) when trip.locationId is null
    - REV-06: Trip.reviewedAt is set to new Date() on successful commit
    - REV-06: Trip.notes is replaced with tripNotes (not appended)
    - If trip.reviewedAt is already set, returns 409 { error: 'Trip already reviewed' }
    - If tripId not found, returns 404
    - Request body missing required fields returns 400
    - All writes happen inside prisma.$transaction — if any step fails, none are committed
    - usageStatus values outside ['used', "didn't need", 'forgot but needed'] return 400
  </behavior>
  <action>
    Write tests first in `tests/review-batch.test.ts` covering the behaviors above (mock prisma client). Then implement `app/api/trips/[id]/review/route.ts`.

    **Request body shape:**
    ```typescript
    {
      gearUsage: Array<{ gearId: string; usageStatus: 'used' | "didn't need" | 'forgot but needed' }>;
      mealFeedbacks: Array<{ mealId: string; mealPlanId: string; mealName: string; rating: 'liked' | 'disliked'; notes?: string }>;
      spotRating: number | null;   // 1-5 int, written to Location.rating
      spotNote: string | null;     // written to Location.notes
      tripNotes: string | null;    // written to Trip.notes (replaces existing)
    }
    ```

    **Endpoint logic:**
    1. Parse tripId from route params
    2. Validate request body — return 400 if gearUsage is not an array, or any usageStatus is invalid
    3. Fetch trip from DB (include locationId) — return 404 if not found
    4. If trip.reviewedAt is already set, return 409
    5. Resolve mealPlanId: for each mealFeedback entry, use the provided mealPlanId (client sends it)
    6. Run prisma.$transaction:
       a. For each { gearId, usageStatus } in gearUsage: `tx.packingItem.update({ where: { tripId_gearId: { tripId, gearId } }, data: { usageStatus } })`
       b. For each mealFeedback: upsert using findFirst({ where: { mealId, mealPlanId } }) pattern from research
       c. If spotRating !== null AND trip.locationId !== null: `tx.location.update({ where: { id: trip.locationId }, data: { rating: spotRating, notes: spotNote ?? undefined } })`
       d. `tx.trip.update({ where: { id: tripId }, data: { notes: tripNotes ?? undefined, reviewedAt: new Date() } })`
    7. Return 200 { success: true, reviewedAt: trip.reviewedAt }

    **Error handling:** wrap entire handler in try-catch, `console.error('Failed to submit trip review:', error)`, return 500 with JSON error.

    **Validation inline** — no external Zod dependency needed given project pattern (see usage/route.ts for the simple manual validation style used elsewhere).

    Note: `PackingItem` has a compound unique key `@@unique([tripId, gearId])` — use `where: { tripId_gearId: { tripId, gearId } }` for the update.
  </action>
