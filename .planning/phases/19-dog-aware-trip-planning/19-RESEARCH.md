# Phase 19: Dog-Aware Trip Planning - Research

**Researched:** 2026-04-02
**Domain:** Prisma schema migration, Next.js API routes, React form state, Claude prompt engineering
**Confidence:** HIGH

## Summary

Phase 19 is a focused vertical slice: one new boolean field on the Trip model propagates through 5 files. The schema migration is trivial — a single nullable Boolean with a default. The API layer needs `bringingDog` threaded through POST (create) and PUT (edit). The component layer gets a checkbox toggle in both the create inline form and the edit Modal. The packing list prompt in `lib/claude.ts` gets a conditional dog section when the flag is true. TripCard gets a 🐕 indicator in the header row.

The phase depends on Phase 15 being complete (it is, per ROADMAP.md and STATE.md). No external dependencies, no new libraries. All patterns for this work already exist in the codebase — the `isWishlist` boolean on GearItem is a direct parallel for the schema field; the emergency contact per-trip fields are a parallel for the API threading pattern; the conditional prompt building in `generatePackingList` follows the same pattern already used for weather section injection.

The main risk area is the packing list regression test: when `bringingDog = false`, no dog items must appear. Claude is non-deterministic, so the regression requirement (DOG-04) is satisfied at the prompt level, not the output level — the prompt must make dog inclusion strictly conditional.

**Primary recommendation:** Single Prisma migration adds `bringingDog Boolean @default(false)` to Trip. Thread the field through POST/PUT routes. Add checkbox to both form surfaces in TripsClient. Update TripCard's TripData interface and header area. Update `generatePackingList` params interface and prompt conditionally.

**Edit form (controlled state):**

### Pattern 5: Conditional Dog Section in Packing List Prompt

The `buildWeatherSection` function is the precedent for conditional prompt sections. When `bringingDog = true`, append to the prompt before the INSTRUCTIONS block:

```typescript
// Source: lib/claude.ts (new conditional, follows buildWeatherSection pattern)
const dogSection = bringingDog
  ? `\nDOG CONTEXT:\nWill is bringing his dog on this trip. Add a "Dog" section to the packing list with essential dog gear: food + collapsible bowl, water bowl, leash + backup leash, poop bags (2x expected amount), dog-specific first aid supplies (tweezers for ticks, wound spray). Note any dog-friendly considerations for the destination.`
  : ''
```

The `generatePackingList` params interface gains `bringingDog?: boolean`. The prompt receives `${dogSection}` interpolated between the weather section and the INSTRUCTIONS block.

The JSON instruction block also needs "dog" added as a valid category name (alongside the existing list: shelter, sleep, cook, power, clothing, tools, vehicle, hygiene, safety, misc).

### Pattern 6: TripCard Dog Indicator

The 🐕 indicator sits in the trip name header row, next to the existing edit/delete buttons area. Parallel: the `isActive` status ribbon and the days countdown are both conditional renders. A simple inline condition after the trip name:

```tsx
{trip.bringingDog && (
  <span title="Bringing dog" className="text-base leading-none">🐕</span>
)}
```

### Anti-Patterns to Avoid

- **Raw boolean pass-through:** Never do `bringingDog: data.bringingDog` — coerce explicitly to `data.bringingDog === true`. Untrusted JSON could pass a truthy string.
- **Mutating existing packing list results:** When `bringingDog` is toggled on an existing trip that already has a persisted `packingListResult`, the cached result is stale (no dog items). The planner should note that changing `bringingDog` on an existing trip does NOT auto-regenerate the packing list — Will must manually regenerate. No automatic invalidation is needed for this phase (out of scope).
- **Duplicating the Dog category enum:** Don't add "dog" as a hardcoded category to `CATEGORY_EMOJIS` unless you also update the JSON instructions. If "dog" is added to the prompt, add it to the CATEGORY_EMOJIS map: `dog: '🐕'`.

## Common Pitfalls

### Pitfall 1: FormData Checkbox Returns "on" or null
**What goes wrong:** `form.get('bringingDog')` returns the string `"on"` when checked and `null` when unchecked — not a boolean. Comparing `=== true` against it will always be false.
**Why it happens:** HTML checkbox FormData serialization.
**How to avoid:** Use `form.get('bringingDog') === 'on'` in the create handler.
**Warning signs:** Dog toggle appears to do nothing in create flow despite being checked.

### Pitfall 2: Edit Modal Opens with bringingDog Always false
**What goes wrong:** `openEdit()` initializes edit form state but doesn't set `editBringingDog`.
**Why it happens:** The new state variable is forgotten in the initialization block.
**How to avoid:** Explicitly set `setEditBringingDog(trip.bringingDog ?? false)` inside `openEdit`.
**Warning signs:** Toggling on an existing trip and saving always resets to false (DOG-05 fails).

### Pitfall 3: TripData Interface Out of Sync
**What goes wrong:** `TripCard` renders with `trip.bringingDog` as `undefined` — TypeScript may not catch it if the interface isn't updated in both files.
**Why it happens:** `TripData` is duplicated in `TripsClient.tsx` and `TripCard.tsx`. Both must be updated.
**How to avoid:** Update both interface definitions in the same task.
**Warning signs:** TypeScript error `Property 'bringingDog' does not exist on type 'TripData'`.

### Pitfall 4: Packing List Category "dog" Not Recognized by CATEGORY_EMOJIS
**What goes wrong:** Dog section renders with the fallback `📦` emoji instead of `🐕`.
**Why it happens:** `CATEGORY_EMOJIS` in `lib/claude.ts` doesn't include `dog`.
**How to avoid:** Add `dog: '🐕'` to `CATEGORY_EMOJIS`.
**Warning signs:** Dog packing section shows box emoji.

### Pitfall 5: Stale Offline Cache After Schema Migration
**What goes wrong:** The PWA offline cache may have old Trip shapes without `bringingDog`. On first load after migration, offline-cached trip objects lack the field.
**Why it happens:** Phase 8 built offline caching; `cachedAt` timestamps are stored in IndexedDB.
**How to avoid:** The `bringingDog` field defaults to `false` in Prisma, so old cached objects lacking the field will simply not show the indicator (safe failure). No cache invalidation is required. Note in the plan that this is a known acceptable degradation.
**Warning signs:** Not a real bug — just no dog indicator on stale cached trips until re-cached.
