## New UI Elements — Interaction Contract

This section specifies the two net-new UI elements Phase 26 adds to the existing depart page. Executor MUST follow these specs exactly.

### 1. Departure Time Row

**Placement:** Immediately below `PageHeader` (trip name + date range), above the `LeavingNowButton`. This is the earliest visible context on the page.

**Layout:** Single row, `flex items-center gap-2 flex-wrap`. Label left, value/input right.

**State A — departure time NOT set:**
```
[label: "Departure time:"]  [amber text: "Not set — times will be relative"]  [datetime-local input]
```
- Label: `text-sm text-stone-500 dark:text-stone-400`
- Amber text: `text-sm text-amber-600 dark:text-amber-400`
- Input: `type="datetime-local"`, styled as existing `Input` component (`px-3 py-2 rounded-lg border border-stone-300 dark:border-stone-600 bg-white dark:bg-stone-800 text-stone-900 dark:text-stone-100 text-sm focus:outline-none focus:ring-2 focus:ring-amber-400 dark:focus:ring-amber-500 focus:border-transparent`)
- Touch target: outer row `min-h-[44px]`

**State B — departure time IS set:**
```
[label: "Departure time:"]  [formatted date: "Fri Apr 18 at 7:00 AM"]  [edit icon button]
```
- Formatted value: `text-sm font-bold text-stone-900 dark:text-stone-100`
- Edit icon: `Pencil` (size 14) from Lucide, `text-stone-400 dark:text-stone-500`, tap opens inline datetime-local input
- Saving: PATCH fires on `blur` or on explicit confirmation — no auto-regen (D-04)

**Saving behavior:**
- On save, call `PATCH /api/trips/[id]` with `{ departureTime: isoString }`.
- On success, update local state. No toast — the formatted value appearing is confirmation.
- On failure: inline `text-sm text-red-600 dark:text-red-400` below the row: "Couldn't save — tap to retry."

### 2. Time Badge on Checklist Items

**Placement:** Right-aligned within the item row, after item text, before the right edge. `ml-auto shrink-0`.

**Render condition:** Only when `item.suggestedTime` is a non-empty string. Items without `suggestedTime` render identically to the current design.

**Visual spec:**
- `text-xs tabular-nums text-stone-400 dark:text-stone-500`
- No background, no border — plain inline text
- Max width: 80px. If time string exceeds this, it wraps (acceptable).
- Checked items: time badge also gets `line-through` to match item text treatment

**Examples of time strings Claude generates:**
- `"9:00 PM Thu"` (night-before task)
- `"6:30 AM"` (morning-of task)
- `"7:00 AM — depart"` (final departure task)
- `"T-30 min"` (relative, when no departure time is set)

## States and Interactions Summary

**Primary focal point:** When the checklist is fully loaded, the primary visual anchor is the first slot heading (`text-lg font-bold`) followed immediately by its checklist items — the eye enters at the top slot and scans down through the task list.

| State | What the user sees |
|-------|--------------------|
| No checklist, no departure time | EmptyState + departure time row in "not set" amber state |
| No checklist, departure time set | EmptyState + departure time row showing formatted time |
| Generating | Skeleton loaders (existing), departure time row still visible above |
| Checklist loaded, no times | Items render without time badges (backwards compat, D-09) |
| Checklist loaded, with times | Items show `suggestedTime` badges right-aligned |
| Item checked | Checkbox amber fill + item text line-through + time badge line-through |
| Offline | Checklist renders from cache; check-offs queue; departure time input disabled |
| Error (generation) | Red banner with "Try Again" link (existing pattern) |
| Error (departure time save) | Inline red text below departure time row |
