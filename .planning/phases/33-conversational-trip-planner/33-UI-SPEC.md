## Net-New Components: Visual Contracts

### 1. TripPlannerSheet

Full-screen overlay. Not a bottom-sheet partial — it covers `inset-0` and goes edge to edge on mobile.

**Container:**
```
fixed inset-0 z-50 flex flex-col
bg-white dark:bg-stone-950
```

**No backdrop/overlay** — it IS the full screen. No animation required (immediate mount/unmount matching existing `if (!open) return null` pattern in `Modal.tsx`). If animation is added, use `animate-slide-up` (already defined in global CSS from `Modal.tsx` usage) on the inner container, not on the fixed wrapper.

**Header bar:**
```
flex items-center justify-between
px-4 py-3
border-b border-stone-200 dark:border-stone-700
bg-white dark:bg-stone-950
sticky top-0 z-10
```

Header left: `"Plan a Trip"` — `text-lg font-semibold text-stone-900 dark:text-stone-50`

Header right (two elements, gap-3):
1. "Add manually" text link — see escape hatch spec below
2. Close button — X icon, 20px, stone-400/600 hover — matches `Modal.tsx` close button exactly

**Body:**
```
flex-1 overflow-hidden
```
Mounts `ChatClient` directly. No additional padding wrapper — `ChatClient` owns its own padding.

**ChatClient height adjustment:** Current `ChatClient` uses `calc(100dvh - 48px - 80px)` to account for nav bars on the `/chat` page. Inside `TripPlannerSheet`, this calculation changes. The sheet body uses `flex-1 overflow-hidden` and `ChatClient` should fill 100% of its parent. The executor must add a `containerHeight` prop or CSS override so `ChatClient` fills `100%` of the flex container instead of using the hardcoded calc.

Recommended: Pass a `fullHeight` prop (boolean) to `ChatClient` — when true, replace the hardcoded `calc(100dvh - 48px - 80px)` style with `height: '100%'`.

### 2. "Add manually" Escape Hatch

Always visible in the sheet header, right side, left of the X button.

```
text-sm          // 14px
text-stone-500 dark:text-stone-400
hover:text-stone-700 dark:hover:text-stone-200
underline
```

No button chrome — naked text link only. Tap target: `py-1 px-1` padding added to the element for touch target compliance (minimum 44px height achieved via parent `py-3` header + text height).

Label: `"Add manually"` — lowercase, no caps, no icon. (Source: D-02, CONTEXT.md)

Behavior: tapping calls `onAddManually` prop → sheet closes → static trip form opens.

### 3. trip_summary Card (inside ChatBubble)

Rendered as a card nested inside the assistant chat bubble `div`, following the identical structural pattern as `deleteConfirm` in `ChatBubble.tsx`.

**Card container** (matches `deleteConfirm` card border pattern, adapted to amber/success tone):
```
mt-3 p-3
border border-stone-200 dark:border-stone-700
rounded-lg
bg-stone-50 dark:bg-stone-900/50
```

**Field rows** inside the card:
- Trip name: `text-sm font-semibold text-stone-900 dark:text-stone-100`
- Date range: `text-sm text-stone-600 dark:text-stone-400` — formatted as "Apr 5 – Apr 7" (use `formatDateRange` from `lib/trip-utils` if accessible, else `toLocaleDateString`)
- Location (if present): `text-xs text-stone-500 dark:text-stone-500` with `MapPin` icon at 12px
- Bringing dog (if true): `text-xs text-stone-500 dark:text-stone-500` — "Bringing the dog"
- Notes (if present): `text-xs text-stone-500 dark:text-stone-500 italic`

**"Create Trip" button:**
```
mt-3 w-full
px-4 py-2.5
text-sm font-medium rounded-lg
bg-amber-600 text-white dark:bg-amber-500 dark:text-stone-900
hover:bg-amber-700 dark:hover:bg-amber-600
transition-colors
disabled:opacity-50
```

Loading state label: `"Creating..."` (disabled, spinner inline — use existing `Button` component `loading` prop if the executor refactors into `<Button variant="primary">`).

**Display text stripping:** The JSON block must be stripped from `displayContent` in `ChatBubble`, exactly as `deleteConfirm` and `recommendations` are stripped. Any prose Claude writes before the JSON block remains visible.

### 4. ChatClient Modifications (minimal)

Two new props added — both backward-compatible with defaults:

| Prop | Type | Default | Purpose |
|------|------|---------|---------|
| `apiEndpoint` | `string` | `'/api/chat'` | Routes requests to `/api/trip-planner` |
| `onTripCreated` | `(tripId: string) => void` | `undefined` | Fires when `message_complete` carries a non-null `tripId` |
| `fullHeight` | `boolean` | `false` | When true, container fills parent instead of `calc(100dvh - 48px - 80px)` |

The `fullHeight` prop changes the root `div` style from `calc(100dvh - 48px - 80px)` to `height: '100%'`. The input bar `sticky bottom-20` class must also be adjusted to `sticky bottom-0` when inside the sheet (no BottomNav below it). Recommend a separate `hideBottomNavOffset` boolean or derive from `fullHeight`.

## Interaction Contracts

### Sheet Open
- Trigger: "Plan Trip" button click in `TripsClient` header
- State: `setShowPlannerSheet(true)`
- Effect: Sheet renders at `z-50` covering the trips page. Page scroll is locked (`document.body.style.overflow = 'hidden'` — already done in `Modal.tsx`; replicate in `TripPlannerSheet`).
- Initial state of chat: One pre-populated assistant message (the greeting). Message is injected via `initialMessages` prop — it is a UI affordance, not DB-persisted.

### Agent Greeting
- Rendered as: An `initialMessages` entry with `role: 'assistant'` and `id: 'greeting'`
- Copy: `"Where are you thinking of going?"` (Claude's discretion — this is the recommended default from RESEARCH.md)
- Visual: Renders as a standard assistant `ChatBubble` (stone background, left-aligned, rounded-2xl rounded-bl-sm)

### Sheet Close
- Via X button: `onClose()` → `setShowPlannerSheet(false)` → scroll lock removed
- Via Escape key: Sheet listens for `keydown` Escape, calls `onClose()`
- Via "Add manually": `onAddManually()` → sheet closes + static form opens
- Via post-creation redirect: `onTripCreated(id)` → `onClose()` → `router.push('/trips/${id}/prep')`

### Summary Card Confirm Flow
1. Agent emits `trip_summary` JSON block in response
2. `ChatBubble` extracts the block via `extractTripSummary(content)`
3. Card renders inside the assistant bubble (below any prose text)
4. User taps "Create Trip" button
5. `onCreateTrip(payload)` fires → `TripPlannerSheet` calls `POST /api/trips` directly
6. On success: `onTripCreated(tripId)` → sheet closes → `router.push('/trips/${tripId}/prep')`
7. On error: Inline error below the card — `"Couldn't create trip. Try again."` (red-500 text, no toast)
8. While creating: Button shows `"Creating..."` with `disabled` state

### Error States (inside chat)
- Network error sending message: `"Something went wrong. Try sending again."` — already handled by `ChatClient` (`setError`)
- Trip creation failure (from confirm button): Inline below the card button — `"Couldn't create your trip. Try again."` in `text-sm text-red-500 dark:text-red-400`
- Stream error: `ChatClient` existing `stream_error` handler fires — same copy as current

## Accessibility

| Concern | Specification |
|---------|---------------|
| Sheet focus | On open, focus moves to the first interactive element in the header (close button or input). Use `autoFocus` on `ChatInput` textarea. |
| Close via Escape | `keydown` listener added on sheet mount — calls `onClose()` |
| Chat scroll region | `role="log"` + `aria-live="polite"` + `aria-relevant="additions"` — already on `ChatClient` scroll div; preserved inside sheet |
| "Add manually" | Plain text link — no icon-only affordance. Always visible, always readable. |
| Create Trip button | `disabled` + `aria-disabled` during loading state |
| Sheet is full-screen | No backdrop to click through. Only close paths are X button, Escape key, Add manually link, and post-creation redirect. |
