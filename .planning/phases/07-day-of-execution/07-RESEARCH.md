# Phase 7: Day-Of Execution - Research

**Researched:** 2026-04-01
**Domain:** Departure checklist generation (Claude AI) + float plan email delivery (Nodemailer/Gmail) + settings persistence
**Confidence:** HIGH

## Summary

Phase 7 adds two features on top of the existing trip prep infrastructure: a Claude-generated time-ordered departure checklist and a safety float plan email. Both follow patterns already established in Phase 6 — AI output generation, Zod parsing, DB persistence, load-on-mount, and regenerate with confirmation. The primary new dependency is Nodemailer for Gmail SMTP delivery.

The checklist is a new Prisma model (DepartureChecklist + items), generated by Claude using packing list, meal plan, power data, and vehicle mods as inputs. Check-off state persists fire-and-forget via a dedicated PATCH endpoint — the same pattern as PackingItem.packed. The float plan is a Claude-written email body, sent via Nodemailer/Gmail App Password, with send events logged to a new FloatPlanLog model. A new Settings model holds the emergency contact (global default + per-trip override on Trip model).

The phase also adds a `/settings` page and a gear icon in `TopHeader`. Three new Prisma models are needed. Nodemailer is not yet installed — it must be added.

**Primary recommendation:** Follow the Phase 6 AI persistence pattern exactly. New models: DepartureChecklist (with items JSON or child rows), FloatPlanLog, Settings. New install: `npm install nodemailer @types/nodemailer`.

### Pattern 1: AI Output Persistence (Phase 6 Pattern — follow exactly)
**What:** Generate → persist to DB → load on mount → regenerate with confirmation
**When to use:** All Claude-generated outputs. DepartureChecklist follows this exactly.
**Example:**
```typescript
// On mount: load saved checklist
useEffect(() => {
  async function load() {
    const res = await fetch(`/api/departure-checklist?tripId=${tripId}`)
    if (res.ok) {
      const data = await res.json()
      if (data.result) setChecklist(data.result)
    }
  }
  load()
}, [tripId])

// Generate: POST to API, persist, set state
async function handleGenerate() {
  setGenerating(true)
  const res = await fetch('/api/departure-checklist', {
    method: 'POST',
    body: JSON.stringify({ tripId }),
  })
  if (res.ok) {
    const data = await res.json()
    setChecklist(data.result)
  }
}
```

### Pattern 2: Fire-and-Forget Check-off Persistence
**What:** Toggle checked state optimistically in UI, PATCH to server without awaiting success
**When to use:** Checklist item check-off. Same as PackingItem.packed in Phase 6.
**Example:**
```typescript
// Source: PackingList.tsx check-off pattern
const handleCheck = useCallback(async (itemId: string, checked: boolean) => {
  // Optimistic update first
  setItems(prev => prev.map(item =>
    item.id === itemId ? { ...item, checked } : item
  ))
  // Fire-and-forget persist
  fetch(`/api/departure-checklist/${checklistId}/check`, {
    method: 'PATCH',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ itemId, checked }),
  }).catch(() => {/* silent fail — user sees local state */})
}, [checklistId])
```

### Pattern 3: Nodemailer Gmail SMTP
**What:** Create transporter with SMTP config from env vars, send HTML email
**When to use:** Float plan send via `/api/float-plan` route
**Example:**
```typescript
// lib/email.ts
import nodemailer from 'nodemailer'

export function createTransporter() {
  return nodemailer.createTransport({
    host: 'smtp.gmail.com',
    port: 587,
    secure: false, // TLS via STARTTLS
    auth: {
      user: process.env.GMAIL_USER,
      pass: process.env.GMAIL_APP_PASSWORD,
    },
  })
}

export async function sendFloatPlan(params: {
  to: string
  toName: string
  subject: string
  html: string
}): Promise<void> {
  if (!process.env.GMAIL_USER || !process.env.GMAIL_APP_PASSWORD) {
    throw new Error('Gmail credentials not configured. Set GMAIL_USER and GMAIL_APP_PASSWORD in .env')
  }
  const transporter = createTransporter()
  await transporter.sendMail({
    from: `"Outland OS" <${process.env.GMAIL_USER}>`,
    to: params.to,
    subject: params.subject,
    html: params.html,
  })
}
```

### Pattern 5: Settings Model (global default + per-trip override)
**What:** Single Settings row holds global defaults; Trip model gets nullable override fields
**When to use:** Emergency contact resolution. API reads Trip.emergencyContactEmail first, falls back to Settings.emergencyContactEmail.
**Data resolution:**
```typescript
// In /api/float-plan route

const settings = await prisma.settings.findFirst()
const recipientEmail = trip?.emergencyContactEmail ?? settings?.emergencyContactEmail
const recipientName  = trip?.emergencyContactName  ?? settings?.emergencyContactName

```

### Anti-Patterns to Avoid
- **Await on check-off:** Never await the PATCH for checkbox persistence — optimistic UI only, same as PackingItem.packed
- **Await on float plan log:** FloatPlanLog write can be fire-and-forget after successful send — don't let log failure block the user response
- **Modal for float plan confirm:** Use ConfirmDialog (not full Modal) per UI-SPEC — simpler UX for a confirmation action
- **Template string for email HTML:** Build the email HTML via Claude composition — don't hand-write a fixed template (D-09)
- **Transporter singleton across requests:** Create transporter per request in Next.js API routes — no module-level singleton (serverless environment)

## Prisma Schema Additions

Three new models are needed. These are Claude's discretion on exact structure but here are the confirmed requirements:

### DepartureChecklist
```prisma
model DepartureChecklist {
  id          String   @id @default(cuid())
  tripId      String   @unique  // one checklist per trip (like MealPlan)
  result      String   // JSON blob: { slots: [{ label, items: [{ id, text, checked }] }], unpackedWarnings: string[] }
  generatedAt DateTime @default(now())
  createdAt   DateTime @default(now())

  trip Trip @relation(fields: [tripId], references: [id], onDelete: Cascade)
}
```
Note: Storing items as JSON in `result` mirrors Trip.packingListResult and MealPlan.result. Check-off state is updated in-place via a PATCH that reads/parses/updates/writes the JSON blob — same cost as a separate table for a single-user app with small checklists.

### FloatPlanLog
```prisma
model FloatPlanLog {
  id          String   @id @default(cuid())
  tripId      String
  sentTo      String   // email address
  sentToName  String?
  subject     String
  body        String   // full email body sent
  sentAt      DateTime @default(now())

  trip Trip @relation(fields: [tripId], references: [id], onDelete: Cascade)

  @@index([tripId])
}
```

### Settings
```prisma
model Settings {
  id                    String  @id @default(cuid())
  emergencyContactName  String?
  emergencyContactEmail String?
  createdAt             DateTime @default(now())
  updatedAt             DateTime @updatedAt
}
```
Note: Single row — no unique constraint needed since the API will upsert on the first record. Trip model also needs two new nullable fields: `emergencyContactName String?` and `emergencyContactEmail String?` for per-trip override (D-08). Add `departureChecklist DepartureChecklist?` relation to Trip.

## Common Pitfalls

### Pitfall 1: Gmail App Password Setup
**What goes wrong:** Nodemailer auth fails with "Invalid login" or "Username and Password not accepted"
**Why it happens:** 2-Step Verification not enabled on the Gmail account, or using account password instead of App Password, or App Password has spaces removed incorrectly
**How to avoid:** Document setup steps clearly in `.env.example`. The 16-character App Password must be pasted with no spaces. 2-Step Verification MUST be enabled first at myaccount.google.com > Security.
**Warning signs:** `535 5.7.8 Username and Password not accepted` error from Gmail SMTP

### Pitfall 2: Nodemailer Transporter as Module Singleton
**What goes wrong:** Works in development, breaks or leaks connections in serverless/edge environments
**Why it happens:** Module-level singleton reuses a connection across requests — not safe in Next.js API routes which may run as serverless functions
**How to avoid:** Create transporter inside the API route handler or in a factory function called per-request (`createTransporter()` pattern above)
**Warning signs:** Intermittent auth errors or connection timeouts after first request succeeds

### Pitfall 3: Check-off State in JSON Blob
**What goes wrong:** Race condition when two rapid check-offs both read the same stale JSON blob, each apply one change, and the second write stomps the first
**Why it happens:** Read-modify-write on a JSON column is not atomic without transactions
**How to avoid:** For a single-user app this is low risk (two simultaneous taps is unlikely), but use Prisma transactions if it becomes an issue. Alternatively, use a separate DepartureChecklistItem table where each item is its own row (direct UPDATE, no read-modify-write).
**Warning signs:** Checkbox "bouncing" back after rapid multiple taps

### Pitfall 4: prepSections Registry Not Updated
**What goes wrong:** Departure summary section doesn't appear on prep page
**Why it happens:** `lib/prep-sections.ts` PREP_SECTIONS array not updated with the new 'departure' entry
**How to avoid:** Update PREP_SECTIONS in the same task that adds the TripPrepClient departure section slot
**Warning signs:** Prep page renders fine but departure section never appears

### Pitfall 5: Missing GMAIL env vars in .env.example
**What goes wrong:** Will has to debug why float plan send fails on a fresh setup
**Why it happens:** New env vars not documented
**How to avoid:** Add `GMAIL_USER=` and `GMAIL_APP_PASSWORD=` to `.env.example` in the same task as `lib/email.ts`

### Pitfall 6: zod v4 API Differences
**What goes wrong:** Build errors like "z.string().email is not a function" or schema import failures
**Why it happens:** Project uses zod v4 (^4.3.6) — some zod v3 patterns changed in v4
**How to avoid:** New Zod schemas (DepartureChecklistResultSchema, FloatPlanEmailSchema) should follow the patterns in `lib/parse-claude.ts` exactly — those already work with zod v4. Email validation: `z.string().email()` works the same in v4.

### Settings API Route Pattern
```typescript
// app/api/settings/route.ts
// GET: return first Settings row (or null)
// PUT: upsert first Settings row (create if none, update if exists)
export async function GET() {
  const settings = await prisma.settings.findFirst()
  return NextResponse.json(settings ?? { emergencyContactName: null, emergencyContactEmail: null })
}

export async function PUT(request: Request) {
  const body = await request.json()
  const existing = await prisma.settings.findFirst()
  if (existing) {
    const updated = await prisma.settings.update({
      where: { id: existing.id },
      data: { emergencyContactName: body.name, emergencyContactEmail: body.email },
    })
    return NextResponse.json(updated)
  } else {
    const created = await prisma.settings.create({
      data: { emergencyContactName: body.name, emergencyContactEmail: body.email },
    })
    return NextResponse.json(created, { status: 201 })
  }
}
```
